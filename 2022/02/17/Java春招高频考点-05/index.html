16. 线程池的作用？为什么需要线程池？创建线程为什么比较耗时？线程池的核心参数有哪些？线程池创建一个线程的时机？

线程池提供了一种限制和管理资源的能力。每个线程池还维护着一些基本统计信息，例如已完成任务的数量。

线程池的好处：

  • 降低资源消耗。通过重复利用已经创建的线程降低线程创建和销毁造成的消耗。
  • 提高响应速度。当任务到达时，任务可以不需要等待的等待线程创建就能立即执行。
  • 提高线程的可管理性。线程时稀缺资源，如果无限制的创建，不仅消耗系统资源，还会降低系统的稳定性，使用线程池可以进行统一的分配，调优和监控。

在线程池的好处中也有提到，线程池就可以重复利用已经创建的线程，如果在线程池中执行一个新的线程，线程池会想着如何复用已有的线程，速度一般都是比直接创建线程快的。同时，当任务到达时，任务可以不需要等待的等待线程创建就能立即执行。所以创建线程比使用线程池要慢。

ThreadPoolExecutor 的源码中已经详细的描述了线程池的核心参数和作用。

线程池的核心参数有：

  • corePoolSize：类型为 int，核心大小，一般情况下线程池中可以同时运行的线程的数量。
  • maximumPoolSize：类型为 int，线程池可以容纳的最大大小，如果等待队列满了的话，maximumPoolsize 会代替 corePoolSize 成为新的 “线程池可以同时运行的线程的数量”。
  • keepAliveTime：类型为 long，如果线程池中线程的数量是大于 corePoolSize 的，多于的空闲线程会等待 keepAliveTime 后被移除。
  • unit：类型为 TimeUnit 枚举类型，是 keepAliveTime 的单位。比如 keepAliveTime 是 1l，TimeUnit 是秒，那么就是等待 1 秒移除。
  • workQueue：类型为BlockingQueue<Runnable> ，阻塞队列。这个并非为正在工作的队列，而是正在等待的队列。官方的解释就是，在这些线程被执行之前去 hold 住他们。
  • handler：类型为RejectedExecutionHandler，是 ThreadExecutor 提供的枚举类型。是在线程池和等待队列都满了的情况下的处理方案。有四种策略，Spring 创建线程池或者我们不指定特定的处理方案的时候，默认为 ThreadPoolExecutor.AbortPolicy。
    • **ThreadPoolExecutor.AbortPolicy**：抛出 RejectedExecutionException来拒绝新任务的处理。
    • **ThreadPoolExecutor.CallerRunsPolicy**：调用执行自己的线程运行任务。您不会任务请求。但是这种策略会降低对于新任务提交速度，影响程序的整体性能。另外，这个策略喜欢增加队列容量。如果您的应用程序可以承受此延迟并且你不能任务丢弃任何一个任务请求的话，你可以选择这个策略。
    • ThreadPoolExecutor.DiscardPolicy： 不处理新任务，直接丢弃掉。
    • ThreadPoolExecutor.DiscardOldestPolicy： 此策略将丢弃最早的未处理的任务请求。
  • threadFactory：类型为 ThreadFactory，在线程池创建一个新线程时使用的工厂。

线程池在池内线程数量小于 corePoolSize 时，会直接在池内创建线程。如果池内线程数量等于 corePoolSize 了，就像需要创建的线程放进 workQueue，在 workQueue 满了之后，就将 maximumPoolSize 作为新的 corePoolSize，只要池内线程数量小于 maximumPoolSize，就可以有序的从 workQueue 中继续创建线程。如果池内继续满了，WorkQueue 也满了，就会启用 Handler，默认是抛出异常拒绝新的创建请求。同时，池内的线程数大于一开始创建的 corePoolSize 时，线程池将多余的空闲线程在 keepAliveTime 的 unit 之后移除，只要有空位置，就可以继续创建线程。

17. Java 中如何创建一个线程？实现 Callable 接口和实现 Runnable 的区别？

创建线程主要要三种方式：

1. 继承 Thread 类创建线程类

  1. 定义 Thread 类的子类，并重写该类的 run 方法，该 run 方法的方法体就代表了线程要完成的任务。

  2. 创建 Thread 子类的实例，即创建了线程对象。

  3. 调用线程对象的 start() 方法启动线程。

2. 通过 Runnable 接口创建线程类

  1. 定义 Runnable 接口的实现类，并重写该接口的 run() 方法，该 run() 方法的方法体同样是该线程的线程执行体。

  2. 创建 Runnable 实现类的实例，并以此实例作为 Thread 的 target 来创建 Thread 对象，该 Thread 对象才是真正的线程对象

  3. 调用线程对象的 start() 方法来启动该线程。

3. 通过 Callable 和 Future 创建线程

  1. 创建 Callable 接口的实现类，实现 call() 方法，该 call 方法作为线程执行体，并且有返回值。
  2. 创建 Callable 实现类的实例，使用 FutureTask 类来包装 Callable 对象，该 FutureTask 对象封装了该 Callable 对象的 call() 方法的返回值。（FutureTask是一个包装器，它通过接受Callable来创建，它同时实现了Future和Runnable接口。）
  3. 使用 FutureTask 对象作为 Thread 对象的 target 创建并启动新线程。
  4. 调用 FutureTask 对象的 get() 方法获得子线程执行之后的返回值。

实现 Callable 接口和 Runnable 接口的区别？

  • Runnable 实现的是 run() 方法，而 Callable 实现的是 call() 方法。
  • Callable 执行后有返回值，而 Runnable 不能有返回值
  • call() 方法可以抛出异常，run() 方法不可以。
  • 运行 Callable 任务可以拿到一个 Future 对象，表示异步计算的结果。它提供了检查计算是否完成的方法，以等待计算的完成，并检索计算的结果。通过 Future 对象可以了解任务执行情况，可取消任务的执行，还可获取执行结果。

18. 创建一个线程时，会在哪些区域分配空间？

在 JVM 创建一个线程的时候，就为其分配一个 Program Counter Register（程序计数器），VM Stack（虚拟机栈）和 Native Method Stack（本地方法栈），当线程终止的时候，三者所占用的内存空间也会被释放掉。

19. Java 中线程间通信的方式有哪些？

1. 利用线程同步实现通信

利用 synchronized 关键字实现线程同步。

由于线程A和线程B持有同一个 MyObject 类的对象 object，尽管这两个线程需要调用不同的方法，但是它们是同步执行的，比如：线程B需要等待线程A执行完了 methodA() 方法之后，它才能执行 methodB() 方法。这样，线程A和线程B就实现了 通信。

2. while 轮询

就是线程 A 通过 while 不停的循环检测线程 B 的某一个状态是否达到。这样线程 A 肯定是可以了解到线程 B 的，但是这样就造成了忙等的现象。没做什么有意义的工作，还加重 CPU 负担。

3. wait / notify 机制

所有类的基类Object 中有 notify() 方法和 wait() 方法。

就比如要一个参数 i = 5 时，线程 A 才能进行进一步操作。当 A 拿到时间片发现 i < 5，A 可以通过 wait() 方法主动放弃时间片，进入阻塞状态。此时 B 线程正在操作 i，当 i = 5 时，B 线程就可以调用 notify() 方法唤醒 A，让它进入可运行状态。这样 CPU 的利用率就提高了。

4. 管道通信

管道是基于“管道流”的通信方式。JDK提供了PipedWriter、 PipedReader、 PipedOutputStream、 PipedInputStream。其中，前面两个是基于字符的，后面两个是基于字节流的。

管道通信的应用场景：

这个很好理解。使用管道多半与I/O流相关。当我们一个线程需要先另一个线程发送一个信息（比如字符串）或者文件等等时，就需要使用管道通信了。

20. 保证线程同步的机制有哪些？

  • 同步方法
  • 同步代码块
  • 使用 ReentrantLock 重入锁实现线程同步
  • 使用 特殊域变量 volatile 实现同步。原理是每次重新计算，安全但并非保证完全一致。
  • 使用原子变量实现线程同步，原理是乐观锁。
  • 使用阻塞队列实现线程同步，BlockingQueue，顺带一提，线程池的 workQueue 就是阻塞队列

21. 什么是 ThreadLocal？实现原理？

ThreadLocal 主要解决的问题是让每个线程绑定自己的值，相当于为线程存放数据的私有盒子。

如果创建了一个 ThreadLocal 变量，那么访问这个变量的每个线程都会有这个变量的本地副本，这也是 ThreadLocal 变量名的由来。他们可以使用 get() 和 set() 方法获取默认值或将其更改为当前线程所存的副本的值，从而避免了线程安全问题。

ThreadLocal 实现原理：

Thread 源码中有一个 ThreadLocals 和一个 inheritableThreadLocals 变量，它们都是 ThreadLocalMap 类型的变量。可以把 ThreadLocalMap 理解为 ThreadLocal 类实现的定制化的 HashMap。默认情况下这两个变量都是 null，只有当前线程调用 ThreadLocal 类的 set 或 get 方法时才创建它们，实际上调用这两个方法的时候，我们调用的就是 ThreadLocalMap 类对应的 get() 和 set()。

ThreadLocalMap是ThreadLocal的静态内部类。

通过上面这些内容，我们足以通过猜测得出结论：最终的变量是放在了当前线程的 ThreadLocalMap 中，并不是存在 ThreadLocal 上，ThreadLocal 只是 ThreadLocalMap 的封装，传递了变量值。ThreadLocal 类中可以通过 Thread.currentThread() 获取到当前线程的对象后，直接通过 getMap(Thread t) 方法访问到该线程的 ThreadLocalMap 对象。

每个Thread中都具备一个ThreadLocalMap，而ThreadLocalMap可以存储以ThreadLocal为 key ，Object 对象为 value 的键值对。

比如我们在同一个线程中声明了两个 ThreadLocal 对象的话， Thread内部都是使用仅有的那个ThreadLocalMap 存放数据的，ThreadLocalMap的 key 就是 ThreadLocal对象，value 就是 ThreadLocal 对象调用set方法设置的值。

22. 什么是 AQS？AQS 有什么用？常见的基于 AQS 的组件有哪些？

AQS 的全称为（AbstractQueuedSynchronizer），这个类在java.util.concurrent.locks包下面。

AQS 是一个用来构建锁和同步器的框架，使用 AQS 能够简单高效地构造出大量应用的泛型同步器，比如 ReentrantLock，Semaphore，其他的诸如 ReentrantReadWriteLock，SynchronousQueue，FutureTask 等等都是基于 AQS的。我们利用 AQS 也能构造出自己需要的同步器。

23. Java 内存模型

说起并发就要说起 Java 的内存模型。Java 内存模型抽象了线程和主内存之间的关系，就比如说线程之间的共享变量必须存储在主内存中。Java 内存模型的主要作用是为了屏蔽系统和硬件的差异，避免一套代码在不同的平台下有不同的作用。

JDK1.2 之前，Java 的内存模型实现总是从 主存 读取变量，是不需要进行特别注意的。而现在，线程可以把变量保存在本地内存（如寄存器）中，而不是直接在主存中进行读写。这样就可能造成一个线程在主存中修改了一个变量的值，另外一个线程还继续使用寄存器中的变量值拷贝，造成数据不一致。

24. 指令重排序是什么？

参考美团技术文档

在实际运行中，代码指令可能并不是严格按照代码语句顺序执行的。像以上代码得到（1，0）、（0，1）、（1，1）都不奇怪。但是得到（0，0）就说明代码指令被重排了。

大多数现代微处理器都会采用将指令乱序执行（out-of-order execution，简称OoOE或OOE）的方法，在条件允许的情况下，直接运行当前有能力立即执行的后续指令，避开获取下一条指令所需数据时造成的等待。通过乱序执行的技术，处理器可以大大提高执行效率。

除了处理器，常见的Java运行时环境的 JIT 编译器也会做指令重排序操作，即生成的机器指令与字节码指令顺序不一致。

25. happens-before规则是什么？作用？有哪些规则？

JMM 在设计的时候，定义了如下策略：

  1. 对于会改变程序执行结果的重排序，JMM 要求编译器和处理器必须禁止这种排序
  2. 对于不回改变程序执行结果的重排序，JMM 对编译器和处理器不做要求。（JMM 允许这样的重排序）

比如这段代码不保证 A happens before B ，因为对结果没有影响。JMM 为了满足编译器和处理器的约束尽可能少，它遵循的规则是：只要不改变程序的执行结果，编译器和处理器想怎么优化怎么优化。

例如，如果编译器经过细致的分析后，认定一个锁只会被单个线程访问，那么这个锁可以被消除。再如，如果编译器经过细致的分析后，认定一个volatile变量只会被单个线程访问，那么编译器可以把这个volatile变量当作一个普通变量来对待。这些优化既不会改变程序的执行结果，又能提高程序的执行效率。

happens-before 规则：

大意是前一个操作的结果对后续操作是可见的。

JMM 为程序员提供的视角就是按照顺序执行的，且满足一个操作 happens-before 于另一个操作，那么第一个操作的执行结果将对第二个执行结果可见，而且第一个操作的执行顺序排在第二个顺序之前。注意，这是 JMM向程序员做出的保证。

但是上面也有说到，JMM 在觉得指令重拍并不会改变执行结构的时候，会尽可能提高程序的并行度。

1. 程序顺序规则

一个线程中，按照程序顺序，前后的操作 Happens-Before 与后续的任意操作。

2. 监视器锁规则

对一个锁的解锁，happens-before 于随后对这个锁的加锁

3. volatile 变量规则

对于 volatile 变量的写操作，Happens-Before 于后续对这个变量的读操作。

4. 传递性

如果 A happens-before B，且 B happens-before C，那么 A happens-before C。

意思是，如果线程 B 能够读到 v = true，那么 x = 42 也是对它可见的。

5. start() 规则

这条是关于线程启动的。它是指主线程 A 启动子线程 B 后，子线程 B 能够看到主线程在启动子线程 B 前的操作。

6. join() 规则

如果线程A执行操作ThreadB.join()并成功返回，那么线程B中的任意操作happens-before于线程A从ThreadB.join()操作成功返回。

26. volatile关键字的作用？实现原理？

volatile 可以防止指令重排序，还可以保证变量的可见行。每次读取这个变量都从主存读取而非寄存器。

编译结束后通过 javap 可以看到 volatile 修饰的变量用 ACC_VOLATILE 修饰，这里就可以供后续操作次变量的时候判断访问标志是否为 ACC_VOLATILE。
